# 10. 一些最佳实践

你已经学会了 Git 的核心操作。本章介绍一些**值得养成的好习惯**。

这些习惯不会让你的代码跑得更快，但会让你的提交历史对他人（包括未来的自己）更加友好。

## 少量多次提交

很多新手倾向于积攒一大堆修改，然后一个 `git commit` 全部提交。

**别这么做。**

五个小提交远比一个大提交好。原因很简单：

- **容易撤销**：如果某个小提交出了问题，`git revert` 可以单独撤销它。一个巨大的提交一旦撤销，你可能丢掉几天的工作。
- **容易 review**：代码审查者更愿意看 5 个清晰的提交，而不是一个"改了一大堆"的提交。

把提交想象成"保存点"——玩得越多，保存得越勤，就越不会丢进度。

## 一个提交只做一件事

这就是所谓的**原子提交**（Atomic Commit）。

每个提交应该代表一个**逻辑上完整的更改**，而不是把不相关的修改混在一起。

举个例子：你修了一个 bug，同时顺手给代码做了格式化。这两件事应该分开提交：

```bash
# 第一次提交：修 bug
git add bugfix.txt
git commit -m "修复用户登录时崩溃的问题"

# 第二次提交：格式化
git add .
git commit -m "统一代码缩进为 4 空格"
```

如果混成一个提交，别人以后想撤销格式化、但保留 bug 修复时，就非常麻烦了。

## 让 AI 帮你检查提交

在没有 AI 的时代，教程会反复强调：**提交前 `git diff` 看一眼**、**提交消息要写得清晰准确**。

这些建议本身没有错，但现在你有了更好的方式：**让 AI 来做这些事。**

- **检查变更**：提交前可以让 AI 帮你 `git diff`，它会告诉你改了什么、有没有不小心留下的调试代码。
- **生成提交消息**：AI 能根据你的实际修改，自动生成一条清晰的提交消息。你只需要确认它写得对不对。

你不需要再担心"我忘了写消息"或者"我写得不够好"——这些都是 AI 擅长的事情。

**但有一个前提：** 你仍然需要理解"好提交"和"坏提交"的区别，才能判断 AI 的建议是否合理。这就是为什么本章仍然要讨论这些概念——不是让你手动执行，而是让你有能力做判断。

## 用独立分支开发新功能

不要把所有的开发都堆在 main 分支上。

每当你开始一个新功能或实验性修改时，先创建一个新分支：

```bash
git switch -c feature-user-login
```

这样做有两个好处：

1. **保持 main 始终可用**：main 分支永远是稳定的版本
2. **实验失败了也没关系**：直接删掉分支就行，不影响 main

功能完成后，再把它合并回 main。这就是专业开发者的日常流程。

## 不要提交敏感信息

**永远不要**把密码、API 密钥、Token 等敏感信息提交到 Git 仓库。

Git 的历史一旦推送到远程（比如 GitHub），这些数据就很难彻底删除——即使你删除了文件，它仍然在提交历史中。

正确做法：

1. 把敏感配置放在 `.env` 文件中
2. 在 `.gitignore` 中添加 `.env`，让 Git 忽略它

```
# .gitignore
.env
*.key
```

> `.env` 只是一个普通文本文件，但绝大多数开发工具（如 Node.js、Python、Rails、Docker Compose 等）都约定俗成地从它读取配置。

如果不小心提交了敏感信息，第一时间**修改密码或密钥**，而不是试图从 Git 历史中删除它。

## 用 Rebase 整理提交历史

`git rebase` 是 Git 中最强大的命令之一，也是很多人害怕使用的命令。

把它理解为一个**历史整理工具**：你开发过程中可能会产生很多琐碎的中间提交（比如"调试中"、"再试试"、"差点好了"），这些细节是你自己开发过程的痕迹，但**其他人不需要关心**。

Rebase 的作用就是把这些细节整理掉，让主干分支的历史变得清晰、简洁。

本章不会深入 rebase 的操作细节，但你只需要知道它存在，它的目的是**让历史对他人更友好**。等你觉得需要时，再去学习它的具体用法。

## 查看某行代码是谁写的

工作中，你经常会遇到一个很自然的需求：**看到某行代码，想知道它是什么时候加的、为什么要这么写、该找谁了解背景。**

这个功能叫 **blame**（追溯）。名字听起来像是"责备"，但它和责备无关——它的真正价值是：当你看到一条奇怪的规则时，能追溯到对应的提交、查看 commit message 了解来龙去脉，必要时还能找到作者进一步了解详情。

本地客户端和编辑器也都有这个功能：
- **TortoiseGit**：右键文件 → "Blame"
- **JetBrains 系列（IDEA、PyCharm 等）**：右键编辑器行号区域 → "Annotate with Git Blame"
- **VS Code**：安装 **GitLens** 扩展后，点击行号旁边就能看到每行的提交信息

在这些客户端里查看 blame，比在终端里用 `git blame` 命令舒服得多——你不仅一眼就能看到上下文，还能直接点击跳到对应的提交，查看当时的 commit message 和完整改动。

## 进阶操作？交给 AI

到目前为止，你已经掌握了 Git 日常开发中最常用的命令。但 Git 还有很多进阶命令：

- `git cherry-pick` — 从别的分支挑选特定的提交
- `git bisect` — 二分查找引入 bug 的提交
- `git stash` — 临时保存未提交的修改
- `git reflog` — 查看包括已删除提交在内的完整历史

**你不需要记住它们。**

这些命令面向的是非常特定的场景，日常开发中很少用到。你只需要知道它们存在就够了——**遇到需要的时候，直接问 AI 就行了**。

这是 AI 时代使用 Git 的最大优势：你不需要死记硬背所有命令，只需要理解核心概念，剩下的交给工具。

## 你刚刚理解了什么

- 少量多次提交比一次大提交更好管理
- 一个提交只做一件事（原子提交）
- 检查变更和写提交消息可以交给 AI，但你需要有能力判断好坏
- 新功能用独立分支开发，保持 main 稳定
- 永远不要提交敏感信息
- Rebase 用于整理历史，让主干历史更清晰
- Blame 帮你追溯每行代码的来源，不是为了甩锅，而是为了找到故事背景
- 进阶命令不需要死记，遇到时问 AI 即可

到这里，你已经完成了整个 Git 入门指南。

回顾一下你走过的路：从创建第一个本地仓库，到学会日常操作、撤销错误、使用分支，再到连接到 GitHub、参与开源协作。你不仅学会了怎么用 Git，更重要的是，你理解了**它为什么这样设计**。

现在，去创建你的下一个仓库吧。
