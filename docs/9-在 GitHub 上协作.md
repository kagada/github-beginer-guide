# 9. 在 GitHub 上协作

上一章，你把本地仓库连接到了 GitHub。现在，让我们进入 GitHub 最核心的价值：**与他人协作**。

## 你可能见过这个流程：Fork → PR

在开源社区，你会经常看到这样的说法："Fork 这个项目，然后提个 PR。"

**Fork** 和 **Pull Request（简称 PR）** 是 GitHub 协作的两个核心概念。

**但别被名字吓到。** Fork 听起来像个高深的概念，但它底层没有任何新的 Git 命令。Fork 本质上就是：

> GitHub 帮你在服务器上执行了一次 `git clone`，把别人的仓库克隆到你的账号下。

你之前学过的所有命令——`clone`、`add`、`commit`、`push`、`switch`、`merge`——在 Fork 的仓库里完全一样地工作。**没有任何新东西要学。**

但问题来了：**原版 Git 可没有 Fork 这个概念。为什么 GitHub 要创造它？**

让我们从头开始理解。

## 原版 Git 的世界：没有权限控制

到目前为止，你接触的所有 Git 命令——`push`、`pull`、`clone`、`merge`——都是原版 Git 就有的功能。但 **Fork 不是**。Fork 是 GitHub 创造的概念。

为什么？因为 Fork 是为了解决一个原版 Git 不需要解决的问题。

想象一下：你在一个纯 Git 的世界里工作。假设你的团队有一个中央仓库，只要有人知道了这个仓库的地址，并且配置了推送权限（比如 SSH 密钥），他就可以：

```bash
git push <仓库地址> main
```

**就这么简单。** 没有任何权限验证，任何人都能直接往中央仓库推送代码。

在早期的开源社区，这就是常态。所有人都是"自己人"，大家互相信任。但当项目变大、参与者变多时，这就成问题了——**任何人都能直接修改别人的项目，这太可怕了。**

## GitHub 的方案：权限管理

GitHub 做了一个根本性的改变：**在 Git 的基础上增加了权限控制，为仓库中的用户定义了两种身份。**

只有拥有以下两种身份的人才能直接 push：
- **仓库所有者**（Owner）：创建仓库的人
- **协作者**（Collaborator）：被所有者邀请的人

如果你既不是所有者也不是协作者——也就是**没有身份**——你连 `git push` 的机会都没有，GitHub 会直接拒绝。

**权限管理解决了安全问题，但也带来了一个新问题：没有身份的人，该怎么贡献代码？**

## Fork 的第一个作用：你自己的副本，随便改

既然你不能直接往别人的仓库 push，那 GitHub 的方案是：**创建你自己的副本，随便改。**

**Fork** 就是别人仓库的一个完整副本，存放在你自己的 GitHub 账号下。你对这个副本有完全的控制权——你想怎么改就怎么改，想怎么 push 就怎么 push，因为这是"你的"仓库。

试试这个：打开任何一个公开的 GitHub 仓库（比如 [git/git](https://github.com/git/git)），点击右上角的 **Fork** 按钮。几秒钟后，你就会在自己的账号下看到一个一模一样的仓库。

**Fork 的本质是：** 一个中间缓冲区。你往这里 push 代码，不需要原仓库的权限。

## Fork 的第二个作用：公网可访问的代码存储

但 Fork 的意义不止于此。我们换个角度想：**就算 owner 想接受你的代码，他怎么拿到？**

在原版 Git 的世界里，如果要合并别人的代码，你得知道对方仓库的地址，然后执行：

```bash
git pull <对方的仓库地址> main
```

**问题来了：** 你的代码在你的本地电脑上。owner 怎么从你的电脑拉取代码？你的本地电脑不一定有公网 IP，还会随时关机，让 owner 直接拉取不现实。

所以 Fork 的第二层含义就出现了：**Fork 把代码放在 GitHub 服务器上，24 小时在线，公网可访问。**

owner 不需要从你的电脑拉取代码，他只需要从你的 **Fork 仓库**（在 GitHub 上）拉取就行了。

**这就是为什么 Fork 是 GitHub 特供：** 它解决了"如何让贡献者的代码随时可被拉取"这个问题。

## Pull Request：请 owner 从你的 Fork 拉取代码

好，你现在 Fork 了一个仓库，在本地 clone 下来，做了修改，push 到你的 Fork。现在你的代码在 GitHub 上，随时可访问。

接下来怎么办？**你得告诉 owner："我的代码准备好了，请来拉取。"**

这就是 **Pull Request（PR）**——字面意思是"拉取请求"。

PR 的逻辑是这样的：

> "我做了一些修改，放在我的 Fork 里了（在 GitHub 服务器上）。**请你（原仓库维护者）从我的 Fork 里拉取这些更改**，如果同意的话。"

**注意方向：** 不是你在 push，而是你在请求别人 pull。owner 可以：
- 查看你改了哪些代码
- 评论、提出问题或建议
- 决定是接受（Merge）还是拒绝

所以 PR 不是一个技术命令，而是一个**协作流程**。它为代码审查和讨论提供了一个结构化的平台。

## 完整的 Fork → PR 流程是怎样的

理解概念后，来看看一次完整的协作通常是怎么进行的：

**1. Fork 一个仓库** — 点击 Fork 按钮，原仓库的副本就出现在你的 GitHub 账号下。

**2. Clone 到本地** — 在你的 Fork 页面点击 **Code** 复制地址，运行 `git clone` 下载到本地。这一步和你之前学过的 `git clone` 没有任何区别。

**3. 创建新分支并修改** — 最佳实践是在新分支上工作，保持 main 分支干净：

```bash
git switch -c my-new-feature
# 做修改
git add .
git commit -m "描述你的修改"
git push -u origin my-new-feature
```

**4. 创建 Pull Request** — 回到 Fork 的 GitHub 页面，点击 **Pull requests** 标签，然后点 **New pull request**。GitHub 会自动对比你的 Fork 和原仓库的差异，填写标题和描述后提交。

提交后，原仓库的维护者会收到通知，他们可以选择合并、评论或关闭。

> **注意：** 如果你是第一次参与开源协作，建议先找一个不活跃的项目或者自己的仓库练手，避免打扰到其他开源贡献者。

## GitHub Issues：讨论的地方

除了 PR，GitHub 还有另一个重要的协作工具：**Issues（议题）**。

Issues 用于讨论：
- 报告 bug（程序哪里出错了）
- 提出新功能建议
- 提问和求助

点击仓库的 **Issues** 标签，然后点击 **New issue** 就能创建。

**PR 和 Issues 的区别**：
- PR 关于代码更改——"我做了这个修改，请合并"
- Issues 关于讨论——"我发现了这个问题"或"我希望能有这个功能"

通常的流程是：先开一个 Issue 讨论想法，达成共识后再通过 Fork 和 PR 提交代码。

## 你刚刚理解了什么

- **原版 Git 没有权限控制**，有地址和凭证就能直接 push
- **GitHub 引入了权限管理**，只有 owner 或协作者能直接 push
- **Fork 有两个作用**：
  1. 你自己的副本——想怎么改就怎么改
  2. 公网可访问的代码存储——owner 随时可以从你的 Fork 拉取
- **Pull Request** 是请求 owner 从你的 Fork 拉取更改
- **Issues** 用于讨论 bug、功能建议和一般问题
- GitHub 的协作模式是：Fork → Clone → 修改 → Push → PR → 讨论 → 合并
