# 第8章：在 GitHub 上协作

上一章，你把本地仓库连接到了 GitHub。现在，让我们进入 GitHub 最核心的价值：**与他人协作**。

## Fork 是什么

假设你看到别人在 GitHub 上分享了一个很棒的项目，你想贡献代码。但你不能直接修改别人的仓库——你没有权限。

**Fork 就是你的解决方案。**

Fork 会创建别人仓库的完整副本，存放在你自己的 GitHub 账号下。你对这个副本有完全的控制权。

试试这个：打开任何一个公开的 GitHub 仓库，点击右上角的 **Fork** 按钮。几秒钟后，你就会在自己的账号下看到一个一模一样的仓库。

## Clone 你 Fork 的仓库

Fork 之后，这个仓库还在 GitHub 服务器上。要修改代码，你需要把它下载到本地。

在 GitHub 上打开你 Fork 的仓库，点击绿色的 **Code** 按钮，复制 HTTPS 地址。然后运行：

```bash
git clone https://github.com/你的用户名/仓库名.git
```

这会下载整个仓库到你当前的文件夹。进入仓库目录，你就可以像在本地仓库一样修改代码、提交更改。

```bash
cd 仓库名
```

## Pull Request 是什么

修改并提交代码后，你希望这些改动能被原仓库采纳。但你不能直接推送到原仓库——怎么把更改"送"回去呢？

**答案就是 Pull Request（简称 PR）。**

Pull Request 是一种请求：**你告诉原仓库的维护者，"我做了一些修改，请考虑合并到你们的项目中。"**

维护者可以看到你的改动，发表评论，决定是否接受。如果接受，你的代码就会成为原仓库的一部分。

## 创建你的第一个 Pull Request

回到 GitHub，打开你 Fork 的仓库页面：

1. 点击顶部的 **Pull requests** 标签
2. 点击绿色的 **New pull request** 按钮
3. GitHub 会对比你的 Fork 和原仓库的差异
4. 点击 **Create pull request**
5. 填写标题和描述，说明你改了什么、为什么这么改
6. 再次点击 **Create pull request** 完成创建

现在，原仓库的维护者会收到通知，看到你的 PR。他们可以选择：
- **Merge**：接受并合并你的更改
- **Comment**：提出问题或建议
- **Close**：关闭 PR，不合并

**PR 不是技术命令，而是协作流程。** 它为代码审查和讨论提供了一个结构化的平台。

## GitHub Issues：讨论的地方

除了 PR，GitHub 还有另一个重要的协作工具：**Issues（议题）**。

Issues 用于讨论：
- 报告 bug（程序哪里出错了）
- 提出新功能建议
- 提问和求助

点击仓库的 **Issues** 标签，然后点击 **New issue** 就能创建。

**PR 和 Issues 的区别**：
- PR 关于代码更改——"我做了这个修改，请合并"
- Issues 关于讨论——"我发现了这个问题"或"我希望能有这个功能"

两者相互配合：通常先开一个 Issue 讨论想法，然后再通过 PR 提交代码。

## 你刚刚理解了什么

- **Fork** 创建别人仓库的副本到你的账号
- **Clone** 下载 Fork 的仓库到本地
- **Pull Request** 请求原仓库维护者合并你的更改
- **Issues** 用于讨论 bug、功能建议和一般问题
- GitHub 的协作模式是：Fork → 修改 → PR → 讨论 → 合并

你刚刚完成了整个 GitHub 入门旅程。从本地仓库到云端协作，你现在理解了 Git 和 GitHub 的核心机制。接下来就是多多练习——在自己的项目中尝试这些操作，参与开源社区，你会越来越熟练。
