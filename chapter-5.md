# 第5章：仓库与仓库之间

上一章，你学会了在同一个仓库里用分支隔离实验。现在，让我们进入下一个层次：**仓库与仓库之间**如何传递代码。

这一章为连接 GitHub 做好了准备——你只需要先在本地理解这些概念。

## 什么是裸仓库？

到目前为止，你用的仓库都有一个工作目录——你能看到 `notes.txt`、`README.md` 这些实际的文件。

**裸仓库**则不同：它没有工作目录，只包含 Git 的内部数据。本质上，它就是普通仓库里 `.git` 文件夹的内容。

裸仓库不能直接编辑文件——它只用来**接收推送**和**提供克隆**。

**GitHub 上托管的就是裸仓库。** 这就是为什么你不能直接在 GitHub 网站上编辑文件（虽然它提供了一些简单的编辑功能，但那只是界面包装）。

## 创建本地裸仓库

让我们用两个本地文件夹来演示仓库之间的操作，而不需要任何网络。

打开 Git Bash，导航到 `my-first-repo` 的**父目录**。然后运行：

```bash
git init --bare my-first-repo.git
```

`--bare` 告诉 Git 创建一个裸仓库。注意目录名以 `.git` 结尾——这是裸仓库的命名惯例。

进去看看里面有什么：

```bash
ls my-first-repo.git
```

你会看到一堆 Git 的内部文件：`HEAD`、`config`、`objects`、`refs` 等。没有 `notes.txt`，没有任何你熟悉的文件。这就是裸仓库的样子。

## 推送你的代码到裸仓库

现在回到你的工作仓库，把它推送到刚才创建的裸仓库。

```bash
cd my-first-repo
```

添加裸仓库作为远程仓库：

```bash
git remote add origin ../my-first-repo.git
```

`git remote add` 给远程仓库起了一个别名。`origin` 是最常用的名字，代表"原始来源"。后面的路径是裸仓库的位置——`../` 表示上一级目录。

**这就是关键区别：**

- 你刚才添加的远程仓库是一个**本地路径**：`../my-first-repo.git`
- 而 GitHub 的远程仓库是一个**网络 URL**：`https://github.com/用户名/仓库名.git`

它们做的事情完全一样，只是位置不同。一个是电脑上的文件夹，一个是互联网上的服务器。

运行这个命令查看远程仓库配置：

```bash
git remote -v
```

你会看到：

```
origin  ../my-first-repo.git (fetch)
origin  ../my-first-repo.git (push)
```

现在推送你的代码：

```bash
git push -u origin main
```

`git push` 把你的提交发送到远程仓库。`-u` 设置上游追踪关系——之后你只需要运行 `git push`，不用再写 `origin main`。

**推送成功了！** 你的代码从工作仓库传输到了裸仓库。

## 从裸仓库克隆

裸仓库已经包含了你的所有提交。现在从它克隆一份，模拟另一个人（或另一台电脑）获取代码。

回到上一级目录并克隆：

```bash
cd ..
git clone my-first-repo.git my-first-repo-copy
```

`git clone` 做了两件事：
1. 下载裸仓库中的所有提交和历史
2. 创建一个带有工作目录的新仓库，让你可以编辑文件

进入克隆的仓库看看：

```bash
cd my-first-repo-copy
git log --oneline
```

所有提交都在这里！打开 `notes.txt`——它的内容和你原来的仓库一模一样。

**克隆是获取远程仓库完整副本的最简单方式。**

## 在克隆仓库中工作并推送

在克隆仓库里试试修改文件：打开 `notes.txt`，加一行内容，比如：

```
这是从克隆仓库添加的内容
```

然后提交：

```bash
git add notes.txt
git commit -m "从克隆仓库添加内容"
```

然后推送回裸仓库：

```bash
git push
```

因为之前设置了上游追踪（`-u`），现在只需要 `git push` 就够了。

回到原来的工作仓库，拉取最新更改：

```bash
cd ../my-first-repo
git pull
```

**`git pull` 从远程仓库获取最新提交并合并到当前分支。** 它相当于 `git fetch` + `git merge` 的组合。

打开 `notes.txt`——克隆仓库的修改已经同步过来了。

## 你刚刚理解了 GitHub

回顾一下你做了什么：

1. 创建了一个裸仓库（没有工作目录）
2. 从工作仓库推送代码到它
3. 从它克隆出一份新的工作仓库
4. 在克隆仓库中修改并推送
5. 在原仓库中拉取最新内容

**GitHub 做的事情和你刚才做的完全一样，只是位置不同。**

对比一下远程仓库的地址：

- 本章你添加的：`../my-first-repo.git` — 一个本地路径
- 下一章连接到 GitHub：`https://github.com/你的用户名/仓库名.git` — 一个网络 URL

命令完全不变：`git push`、`git pull`、`git clone`。Git 不在乎远程仓库是在你硬盘上的另一个文件夹，还是在地球另一端的服务器上。

理解了这一点，连接到 GitHub 就不再有神秘感了——你只是在把同一个操作，从本地路径换成网络地址而已。

## 你刚刚学会了什么

- 裸仓库 — 没有工作目录，只包含 Git 数据
- `git init --bare` — 创建裸仓库
- `git remote add <name> <path>` — 添加远程仓库
- `git push -u origin main` — 推送到远程
- `git clone` — 从远程仓库克隆
- `git pull` — 拉取远程更改

## 常见问题：我能直接用 `.git` 文件夹当裸仓库吗？

你可能会想：既然裸仓库就是 `.git` 文件夹的内容，那我能不能把别的仓库推送到自己的 `.git` 文件夹里？

这是一个很自然的好想法——既然它们内容一样，何必多此一举呢？

**但是不行。** Git 不允许一个目录既作为工作目录的内部管理目录，又作为接收推送的目标仓库。

简单说：`.git` 是 Git 用来管理当前工作区的"私人家当"，不能同时拿来当别人推送的目的地。这就是为什么你需要用 `git init --bare` 单独创建一个裸仓库——它是一个独立的、专门用来接收推送的地方。
